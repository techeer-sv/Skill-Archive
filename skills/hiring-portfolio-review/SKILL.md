# 포트폴리오 리뷰 (Hiring Portfolio Review)

포트폴리오의 기술적 깊이, 문제 해결 과정, 아키텍처 의사결정 품질을 평가합니다.

---

## 핵심 판정 기준 (인라인)

```
[PASS]       - 3축 모두 B 이상, 1개 이상 A
[NEEDS_WORK] - 3축 중 1개 이상 C, 또는 전반적으로 B이나 A 없음
[REJECT]     - 3축 중 1개 이상 D, 또는 2개 이상 C

3축: 문제 인식 / 해결 과정 / 성과·임팩트 (각 A/B/C/D)
등급 판정 시 포트폴리오 원문 인용 필수.
```

---

## 실행 절차

### Step 0: 평가 프레임워크 로딩

다음 파일을 Read 도구로 읽고 내용을 숙지한 후 진행한다 (프로젝트 내 또는 `~/.claude/skills/` 경로):
- `hiring-common/evaluation-framework.md` — 3축 평가 체계, 등급 기준, 경력 수준별 캘리브레이션
- `hiring-common/input-parser.md` — 입력 파싱 및 에러 처리 가이드 (선택)

> 파일을 읽을 수 없는 경우, 위의 "핵심 판정 기준"만으로도 진행 가능하다.

### Step 1: 입력 수집

**포트폴리오 (필수):**
- "리뷰할 포트폴리오 파일 경로를 알려주세요. (PDF, Markdown, 또는 프로젝트 디렉토리)"
- 미제공 시: "포트폴리오는 필수입니다. 파일 경로를 알려주세요." 출력 후 대기

**채용 공고 (선택):**
- "채용 공고가 있으면 내용을 붙여넣어주세요. URL도 가능합니다. (없으면 '없음')"
- URL 제공 시 WebFetch 시도. 실패하면 직접 붙여넣기 요청.

### Step 2: 포트폴리오 파싱

입력 형식에 따라:
- **파일 (PDF/MD/TXT)**: `Read` 도구로 읽기
- **디렉토리**: `Glob`으로 구조 파악 → README 및 주요 파일 `Read`

포트폴리오에서 식별할 항목:
- 프로젝트 목록
- 각 프로젝트의 기술 스택, 역할, 기여도
- 아키텍처/설계 관련 서술
- 경력 수준 추정

### Step 3: 프로젝트별 심층 평가

포트폴리오는 이력서보다 상세하므로 3축 평가의 **"품질"**을 중심으로 평가한다.

#### 3-1. 프로젝트 선정 이유
- 왜 이 프로젝트를 포함했는가?
- 기술적 범위의 다양성이 적절한가?

#### 3-2. 문제 인식 (축 1) — 품질 평가

각 프로젝트에 대해:
- 근본 문제를 정확히 정의했는가?
- First Principles: 기존 가정에 도전했는가? 문제를 근본부터 분해했는가?
- 비즈니스/사용자 관점에서의 임팩트가 서술되어 있는가?

#### 3-3. 해결 과정 (축 2) — 품질 평가

**아키텍처/설계 품질:**
- 시스템 아키텍처가 문제에 적합한가?
- 확장성, 유지보수성을 고려했는가?

**트레이드오프 분석:**
각 주요 의사결정에 대해:
- 기술 스택 선택: 왜 이 기술인가? 대안은?
- 아키텍처 패턴: 장단점 인식이 있는가?
- 비용/효과: 투입 리소스 대비 합리적인가?

> 포트폴리오에서는 트레이드오프 분석이 구체적으로 기술되어 있을 수 있다.
> 이 경우 이력서보다 엄격하게 품질을 평가한다.

#### 3-4. 성과/임팩트 (축 3) — 품질 평가
- 정량적 지표가 있는가? 맥락이 있는가?
- 개선 이터레이션이 있었는가?
- 본인 기여도가 명확한가?

### Step 4: 포트폴리오 전체 구성 평가

개별 프로젝트를 넘어:
- 기술적 범위의 다양성
- 난이도 분포
- 스토리텔링 (프로젝트 간 성장이 보이는가)
- 포지션 정렬도 (공고 제공 시)

### Step 5: 결과 출력

```markdown
# 포트폴리오 리뷰 결과

## 종합 판정: [PASS / NEEDS_WORK / REJECT]
경력 수준 기준: [주니어/미드/시니어]

### 평가 요약
| 평가 축 | 등급 | 핵심 근거 |
|---------|------|----------|
| 문제 인식 | [A/B/C/D] | "[원문 인용]" — [분석] |
| 해결 과정 | [A/B/C/D] | "[원문 인용]" — [분석] |
| 성과/임팩트 | [A/B/C/D] | "[원문 인용]" — [분석] |

### 프로젝트별 상세 분석

#### [프로젝트명]
- **문제 인식**: [등급] — "[인용]" — [분석]
- **해결 과정**: [등급] — "[인용]" — [분석]
  - 트레이드오프: [핵심 의사결정 분석]
- **성과**: [등급] — "[인용]" — 신뢰도: [충분/부분적/불충분]

### 포트폴리오 구성 평가
- 다양성: [평가]
- 난이도 분포: [평가]
- 스토리텔링: [평가]

### 강점
- [강점]: "[근거 인용]"

### 개선 필요 사항
- [개선점]: [구체적 개선 방법]

### 포트폴리오 재구성 제안
[프로젝트 추가/제거/순서 변경/내용 보강 제안]
```

## 주의사항

- 프로젝트의 규모나 유명세로 판단하지 않는다. 사고 과정과 의사결정 품질을 본다
- 코드 리뷰가 아니다. 문제 해결 과정과 기술적 의사결정을 평가한다
- 확인 불가 성과는 "근거 불충분"으로 표기하되 부정적으로 추측하지 않는다
