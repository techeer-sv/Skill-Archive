# 포트폴리오 리뷰 (Hiring Portfolio Review)

포트폴리오를 실제 채용 담당자/기술 리드 관점에서 평가하고 개선 피드백을 제공합니다.

---

## 판정 기준 (인라인)

```
[PASS]   — 프로젝트 품질 3개 영역 모두 "적합" 이상, 포지션 정렬도 충분
[REJECT] — 핵심 영역 1개 이상 "부적합" OR 포지션과의 관련성 부족
```

---

## 실행 절차

### Step 0: 평가 프레임워크 로딩 (선택)

다음 파일을 Read 도구로 읽는다 (프로젝트 내 또는 `~/.claude/skills/` 경로):
- `hiring-common/evaluation-framework.md`
- 읽을 수 없어도 위의 인라인 판정 기준으로 진행 가능

### Step 1: 입력 수집

**포트폴리오 (필수):**
- "리뷰할 포트폴리오 파일 경로를 알려주세요. (PDF, Markdown, 또는 프로젝트 디렉토리)"
- 미제공 시: "포트폴리오는 필수입니다. 파일 경로를 알려주세요." 출력 후 대기

**채용 공고 (선택):**
- "채용 공고가 있으면 내용을 붙여넣어주세요. URL도 가능합니다. (없으면 '없음')"
- URL 제공 시 WebFetch 시도. 실패하면 직접 붙여넣기 요청.

### Step 2: 포트폴리오 파싱

입력 형식에 따라:
- **파일 (PDF/MD/TXT)**: `Read` 도구로 읽기
- **디렉토리**: `Glob`으로 구조 파악 → README 및 주요 파일 `Read`

포트폴리오에서 식별할 항목:
- 프로젝트 목록
- 각 프로젝트의 기술 스택, 역할, 기여도
- 아키텍처/설계 관련 서술
- 경력 수준 추정

### Step 3: 프로젝트별 평가

각 프로젝트를 3개 영역으로 평가한다.

#### 영역 1: 기술적 의사결정 품질

실제 채용 담당자가 가장 중시하는 부분. "왜 이 기술/방법을 선택했는가?"

- 기술 스택 선택 근거가 있는가?
- 대안을 검토한 흔적이 있는가?
- 트레이드오프를 인식하고 있는가?
- 과도한 엔지니어링 없이 적정 수준인가?

| 등급 | 기준 |
|------|------|
| 우수 | 선택 근거 + 대안 비교 + 트레이드오프 명시 |
| 적합 | 합리적 선택이나 대안 비교 부족 |
| 부적합 | 선택 근거 없음 또는 부적절한 기술 선택 |

#### 영역 2: 문제 해결 과정

프로젝트가 해결한 문제의 정의와 접근 방식.

- 해결하려는 문제가 명확히 정의되어 있는가?
- 문제 분해와 접근 과정이 서술되어 있는가?
- 제약 조건을 올바르게 반영했는가?

| 등급 | 기준 |
|------|------|
| 우수 | 문제 정의 + 체계적 접근 + 제약 반영 |
| 적합 | 문제 해결은 했으나 과정 서술 부족 |
| 부적합 | 문제 정의 불명확 또는 접근 방식 비논리적 |

#### 영역 3: 성과와 임팩트

실제 채용에서 정량적 성과가 있는 이력서는 2.5배 더 면접 기회를 얻는다.

- 정량적 성과가 있는가? (수치 + 맥락)
- 본인 기여도가 명확한가?
- 프로젝트의 실제 사용/배포 여부

| 등급 | 기준 |
|------|------|
| 우수 | 정량적 성과 + 맥락 + 본인 기여 명확 |
| 적합 | 성과는 있으나 정량적 근거 부족 |
| 부적합 | 성과 서술 없음 또는 모호함 |

### Step 4: 포트폴리오 전체 구성 평가

개별 프로젝트를 넘어:
- **기술적 범위**: 다양한 기술/도메인을 커버하는가?
- **난이도 분포**: 도전적인 프로젝트가 있는가?
- **포지션 정렬도** (공고 제공 시): JD와 프로젝트가 관련 있는가?
- **스토리텔링**: 프로젝트 간 성장이 보이는가?

### Step 5: 결과 출력

```markdown
# 포트폴리오 리뷰 결과

## 종합 판정: [PASS / REJECT]
경력 수준 기준: [주니어/미드/시니어]

### 프로젝트별 평가

#### [프로젝트명]
| 영역 | 등급 | 근거 |
|------|------|------|
| 기술적 의사결정 | [우수/적합/부적합] | "[인용]" — [분석] |
| 문제 해결 과정 | [우수/적합/부적합] | "[인용]" — [분석] |
| 성과/임팩트 | [우수/적합/부적합] | "[인용]" — [분석] |

[반복...]

### 포트폴리오 구성 평가
- 기술적 범위: [평가]
- 난이도 분포: [평가]
- 포지션 정렬도: [평가]

### 강점
- [강점]: "[근거 인용]"

### 개선 필요 사항
- [개선점]: [구체적 개선 방법]

### 포트폴리오 재구성 제안
[프로젝트 추가/제거/순서 변경/내용 보강 제안]
```

## 주의사항

- 프로젝트의 규모나 유명세로 판단하지 않는다. 의사결정 품질과 사고 과정을 본다
- 코드 리뷰가 아니다. 기술적 의사결정과 문제 해결 과정을 평가한다
- 확인 불가 성과는 "근거 불충분"으로 표기하되 부정적으로 추측하지 않는다
- REJECT여도 구체적 개선 방향을 반드시 제시한다
