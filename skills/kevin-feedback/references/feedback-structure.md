# 이력서 피드백 구조 가이드

이 파일은 이력서 피드백을 작성할 때 따라야 할 구조를 정의합니다.
모든 섹션의 텍스트는 분석된 대상 유저의 말투로 작성합니다.

---

## 피드백 구조

### 1. 첫인상 (전체적 인상)

- 이력서를 처음 봤을 때의 인상을 **한두 문장**으로 표현
- 긍정적/중립적/개선 필요 여부를 솔직하게 말투로 전달
- 예시 분위기: "오, 전반적으로 정리는 잘 됐는데 뭔가 임팩트가 좀 아쉬운 것 같아요~"

---

### 2. 프로젝트별 피드백

경험/프로젝트 **각 항목마다** 아래 4가지를 한 묶음으로 작성:

#### 잘 된 점
- 구체적으로 어떤 부분이 좋은지
- 기술 도입 이유, 문제 정의, STAR 구조 관점에서 칭찬

#### 아쉬운 점
- 빠진 수치, 불명확한 결과, 약한 서술 지적

#### 냉정 분석
면접관/시니어 개발자 시각으로 논리적 허점을 날카롭게 파고드는 파트.
말투는 Kevin 스타일 유지하되, 톤은 직설적으로.

분석 관점:
- **수치의 신뢰성**: 측정 기준 불명확, 환경 미표기, 자의적 조건
- **기술 선택 타당성**: 오버엔지니어링, 더 단순한 해결책이 있지 않은지
- **인과관계 오류**: "A를 했더니 B가 됐다"가 실제 인과인지 상관인지
- **주장-증거 불일치**: 문제 정의만 있고 결과 없는 경우, 반대도 마찬가지
- **실제 규모 의심**: 사용자 수/트래픽 없이 고가용성 패턴 적용한 경우
- **검증 불가 주장**: 측정 불가한 성과 서술

#### 이렇게 고쳐봐요
해당 프로젝트에 대한 구체적 수정 제안 1~3개.
- "~하면 좋을 것 같아요~", "~로 바꿔보는 거 어때요~~?" 형태

---

### 3. 스킬셋 피드백

- **적절성**: 지원 직무/목표에 맞는 스킬인지
- **빠진 스킬**: 보완하면 좋을 기술 스택 또는 역량
- **과도한 나열**: 불필요하게 많은 스킬을 나열하고 있지 않은지

---

### 4. 총평 (한 줄 요약)

- 이력서 전체를 한 문장으로 평가
- 해당 유저의 특징적인 표현 방식으로 마무리
- 예시 분위기: "전체적으로 잠재력은 보이는데 임팩트 있는 숫자가 더 필요해요, 조금만 다듬으면 훨씬 좋아질 것 같아요~"

---

## 출력 형식 예시

### 일반 모드 (`/kevin-feedback`)

```
## 이력서 피드백 by Kevin

**첫인상**
[말투로 작성된 전체 인상]

---

**[프로젝트명 1]**

✅ 잘 된 점
[내용]

😅 아쉬운 점
[내용]

🔍 냉정 분석
[논리적 허점]

💡 이렇게 고쳐봐요
1. [제안]

---

**[프로젝트명 2]**
...

---

**스킬셋**
[스킬셋에 대한 의견]

**총평**
[한 줄 요약]
```

---

### 냉정 모드 (`/kevin-feedback-cold`)

```
## 이력서 피드백 by Kevin (냉정 모드)

**첫인상**
[한 문장 직접 평가]

---

**[프로젝트명 1]**

✅ 잘 된 점
[1~2줄, 사실 확인 수준]

❌ 문제점
[논리 허점, 수치 신뢰성, 인과관계 오류 등 직격]

🔧 수정 방향
1. [액션 중심 제안]

---

**[프로젝트명 2]**
...

---

**스킬셋**
[불필요한 것, 빠진 것 직접 지적]

**총평**
[합격 가능성 관점의 한 줄 평가]
```
