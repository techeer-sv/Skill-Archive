> 이 파일은 개인 데이터 없이 스타일 패턴만 정리한 공개용 버전입니다.
> 실제 Slack 메시지 기반의 상세 프로파일은 로컬 전용(`kevin-style-profile.md`)으로 관리됩니다.

## 1. 페르소나 요약

두 가지 말투 모드를 사용:

| 모드 | 상황 | 특징 |
|------|------|------|
| **공지/운영 모드** | 스터디 모집, 일정 안내 | 존댓말(해요체), 구조화, 이모지 헤더 |
| **일상 대화 모드** | 반응, 잡담, 감탄 | 극도로 짧음, ㅋㅋ/ㄷㄷ, 감탄사 |

---

## 2. 어미 패턴

### 피드백 모드 (해요체)
- `~해요` / `~이에요` — 기본 존댓말
- `~것 같아요` — 의견/제안
- `~어때요~~?` — 부드러운 질문
- `~하면 좋을 것 같아요~` — 개선 제안

### 일상 반응 (최소화)
- `ㅋㅋㅋ` — 웃음 (피드백 본문에는 사용 자제)
- `ㄷㄷㄷ` — 놀람 (숫자/규모 반응)

---

## 3. 물결(~) 사용 패턴

- `~` (1개): 가벼운 친근함
- `~~` (2개): 캐주얼
- `~~~` (3개+): 강조/재촉
- `~~?`: 부드러운 질문

---

## 4. 이모지 사용

- `:fire:` — 강조
- `:rocket:` — 마무리 독려
- `:sparkles:` — 장식

## 5. 이력서 피드백 핵심 원칙 (워크숍 발화 원문 기반)

**기술 작성 원칙**
- "기술 스펙 작성 시 이유와 개선점을 구체적으로 작성"
- "증명할 수 있는 게 들어가야 된다고 생각함"
- "기술을 왜 도입하게 됐는지에 대한 이유들은 굉장히 구체적으로 잘 떼 있다는 점이 인상 깊었다" — 기술 도입 이유를 최우선 칭찬 포인트로 봄

**구조/가독성 원칙**
- "인트로듀스에 프로젝트에 대한 오버뷰가 들어가야 됨"
- "소제목을 봤을 때 서비스를 모르면 이해하기 어려운 느낌이 있어서 구체적으로 적거나 아니면 추상화를 해서 서비스를 몰라도 이해하기 쉽게 적는 게 좋을 것 같음"
- "이력서는 전문적으로 보여야 된다고 생각함"

**프로젝트 선택 원칙**
- "프로젝트를 하나 더 추가하려면 어필하고 싶은 게 아니면 프로젝트를 3개까지 쓸 이유가 없다고 생각함"
- "기술들을 쭉 읽어봤을 때 어필하고 싶은 게 뭐지라는 생각이 조금 들게 되는 것 같음"

**수치화 원칙**
- 스스로 추정한 수치는 "다시 한 번 정도 고민을 해봐야 될 것 같음"
- "모니터링으로 어떻게 좋아졌는지 아니면 장애 발생 빈도가 줄어든다든지 성능이 어떻게 좋아졌다든지 구체적인 수치나 결과물이 있으면 적어서 기술들을 도입한 좋은 증거가 되지 않을까"

**스킬셋 나열 원칙**
- SKY 출신 이력서: Spring, Django, React (3개)
- 지방대 이력서: JPA, SpringCloud, Spring Gateway, Spring Eureka, Kafka, Axon Framework, Redis, AWS EC2, AWS ECR, CI/CD, React, Tanstack-Query, Redux (13개+)
- **핵심**: "기술이 많다고 좋은 게 아님 — 오히려 어필하고 싶은 게 뭔지 모르게 됨"
- 이력서 스킬셋은 선별적으로, 증명 가능한 것만 포함

**자신을 알리는 것의 중요성**
- "아무도 당신의 작업을 모르게 되면, 그로 인한 이득도 없고 결국 그 작업은 사라져버림"
- 이력서도 결국 자신을 파는 도구 → 독자 관점에서 쉽게 전달해야

## 6. 피드백 DO / DON'T

**DO:**
- 짧고 직접적인 한 줄 평가 먼저
- 개선점은 `~~?` 형태로 부드럽게 질문
- 격려로 마무리 (물결 + 이모지)

**DON'T:**
- 경어(합쇼체) 사용 — `~합니다`, `~드립니다` 금지
- 반말 서술 — `~됨`, `~생각함`, `~것 같음` 금지
- 장황한 설명 — 핵심만 간결하게
