# Hiring Critic 평가 프레임워크

> First Principles 사고 기반의 체계적 평가 시스템.
> 모든 채용 스킬에서 이 프레임워크를 참조하여 일관된 평가를 수행한다.

---

## 평가 철학

**"왜?"에서 시작하라.** 모든 경험, 프로젝트, 의사결정을 표면적 결과가 아닌 근본 원리부터 역추적하여 평가한다.

핵심 원칙:
- 결과보다 사고 과정을 본다
- 화려한 기술 스택보다 문제-해법의 논리적 연결을 본다
- 정량적 성과의 "맥락"을 반드시 확인한다
- 트레이드오프를 인식하고 의사결정 근거를 설명할 수 있는지 본다

---

## 3축 평가 체계

> 이 3축 평가는 **이력서와 포트폴리오의 프로젝트 경험 평가**에 최적화되어 있다.
> 코딩 테스트, 면접 등 다른 스킬에서는 참고 프레임워크로만 사용하고, 각 스킬 고유의 평가 체계를 우선한다.

### 축 1: 문제 인식 (Problem Recognition)

**핵심 질문: "올바른 문제를 풀고 있는가?"**

#### First Principles 체크리스트
- [ ] 문제의 근본 원인(root cause)을 정확히 식별했는가?
- [ ] 증상(symptom)과 원인(cause)을 구분하고 있는가?
- [ ] "왜 이 문제가 중요한가"에 대한 명확한 답이 있는가?
- [ ] 문제의 범위(scope)와 영향 범위(blast radius)를 파악했는가?
- [ ] 기존 가정(assumption)을 무비판적으로 수용하지 않았는가?
- [ ] 문제를 더 작은 하위 문제로 분해할 수 있는가?

#### 평가 등급
| 등급 | 기준 |
|------|------|
| **A** | 근본 원인을 정확히 식별하고, 문제의 본질을 자신의 언어로 재정의. 기존 가정에 도전. |
| **B** | 문제를 올바르게 이해하나, 깊이가 부족하거나 일부 가정을 그대로 수용. |
| **C** | 표면적 증상만 다루며, 근본 원인 분석이 없음. |
| **D** | 문제 자체를 잘못 정의하거나, 해결할 필요 없는 문제를 풀고 있음. |

---

### 축 2: 해결 과정 (Solution Process)

**핵심 질문: "최적의 방법으로 풀었는가?"**

#### First Principles 체크리스트
- [ ] 가능한 대안(alternatives)을 충분히 탐색했는가?
- [ ] 선택한 접근법의 근거가 논리적인가?
- [ ] 트레이드오프를 명확히 인식하고 의도적으로 선택했는가?
- [ ] 제약 조건(constraints)을 올바르게 반영했는가?
- [ ] 과도한 엔지니어링(over-engineering) 없이 적정 수준의 복잡도인가?
- [ ] 해결 과정이 재현 가능하고 설명 가능한가?

#### 트레이드오프 분석 매트릭스

아래 형식으로 의사결정을 분석한다:

```
┌─────────────────────────────────────────────────────────┐
│ 의사결정 포인트: [예: 데이터베이스 선택]                      │
├──────────────┬──────────────┬───────────────────────────┤
│ 선택지        │ 장점         │ 단점                       │
├──────────────┼──────────────┼───────────────────────────┤
│ 선택한 방법   │ ...          │ ...                       │
│ 대안 A       │ ...          │ ...                       │
│ 대안 B       │ ...          │ ...                       │
├──────────────┴──────────────┴───────────────────────────┤
│ 선택 근거: [왜 이 방법을 선택했는가]                         │
│ 수용한 리스크: [어떤 단점을 감수했는가]                       │
└─────────────────────────────────────────────────────────┘
```

#### 평가 등급
| 등급 | 기준 |
|------|------|
| **A** | 다수의 대안을 비교하고, 트레이드오프를 명확히 설명하며, 투입 대비 산출이 합리적. |
| **B** | 합리적인 접근이나, 대안 비교가 부족하거나 트레이드오프 인식이 약함. |
| **C** | 한 가지 방법만 시도. 왜 이 방법인지 설명 부족. |
| **D** | 비효율적이거나 부적절한 방법. 과도한 엔지니어링 또는 과소 설계. |

---

### 축 3: 성과/임팩트 (Outcome & Impact)

**핵심 질문: "실제로 의미 있는 변화를 만들었는가?"**

#### First Principles 체크리스트
- [ ] 성과가 구체적이고 측정 가능한가?
- [ ] 성과 수치의 맥락(baseline, 비교 대상)이 제시되었는가?
- [ ] 본인의 기여(contribution)와 팀/외부 요인이 구분되는가?
- [ ] 단기적 성과뿐 아니라 장기적 영향도 고려했는가?
- [ ] 성과가 처음 정의한 문제를 실제로 해결했는가?

#### 성과 신뢰도 판정

| 신뢰도 | 기준 | 예시 |
|--------|------|------|
| **근거 충분** | 정량적 수치 + 맥락 + 측정 방법 명시 | "응답 시간을 2초→0.3초로 개선 (p95, New Relic 기준)" |
| **부분적 근거** | 수치는 있으나 맥락 또는 측정 방법 불명확 | "성능 50% 개선" |
| **근거 불충분** | 주관적 서술 또는 수치 없음 | "성능이 크게 개선됨" |

#### 평가 등급
| 등급 | 기준 |
|------|------|
| **A** | 근거 충분한 정량적 성과 + 문제 해결 확인 + 장기적 영향 고려. |
| **B** | 성과가 있으나 부분적 근거. 또는 정성적 성과만 제시. |
| **C** | 성과 서술이 모호하거나, 문제 해결과의 연결이 약함. |
| **D** | 성과 없음 또는 오히려 부정적 결과. |

## 스킬별 평가 적용 가이드

| 스킬 | 3축 평가 적용 | 고유 평가 체계 |
|------|-------------|--------------|
| 이력서 리뷰 | **전면 적용** (존재 여부 중심) | JD 매칭 분석 |
| 포트폴리오 리뷰 | **전면 적용** (품질 중심) | 아키텍처/설계 품질 |
| 코딩 테스트 | 참고 프레임워크 | **고유 점수 체계 우선** (정확성, 복잡도, 코드 품질) |
| 면접 (기술) | 참고 프레임워크 | **고유 평가 우선** (기술 깊이, 의사결정, 커뮤니케이션) |
| 면접 (인성) | 미적용 | **STAR 프레임워크 우선** (상황-과제-행동-결과) |

## 경력 수준별 기대치 캘리브레이션

| 등급 | 주니어 (0~3년) | 미드 (3~7년) | 시니어 (7년+) |
|------|---------------|-------------|-------------|
| **A** | 문제를 명확히 이해하고 체계적으로 접근 | 대안 비교와 트레이드오프 인식 | 비즈니스 맥락과 장기적 영향까지 고려 |
| **B** | 기본적 문제 해결 능력 발휘 | 합리적 의사결정이나 깊이 부족 | 기술적으로 우수하나 전략적 시야 부족 |
| **C** | 학습 의지는 있으나 경험 부족 | 경력에 비해 깊이가 얕음 | 시니어에 기대되는 리딩 역량 부족 |

## 종합 판정 체계

### 판정 등급 (이력서/포트폴리오용)

```
[PASS]       - 3축 모두 B 이상, 1개 이상 A
[NEEDS_WORK] - 3축 중 1개 이상 C, 또는 전반적으로 B이나 A 없음
[REJECT]     - 3축 중 1개 이상 D, 또는 2개 이상 C
```

### 등급 판정 시 필수 규칙

1. **원문 인용 필수**: 등급을 부여할 때 반드시 이력서/포트폴리오의 **해당 문구를 직접 인용**하여 근거로 제시해야 한다
2. **확인 불가 항목 인정**: 이력서에 없는 정보로 추측하지 않는다. "이 항목에 대한 정보가 이력서에 없으므로 평가 불가"로 표기한다
3. **경력 수준 명시**: 판정 시 어떤 경력 수준 기준을 적용했는지 명시한다

### 출력 형식

```markdown
## 종합 판정: [PASS / NEEDS_WORK / REJECT]
경력 수준 기준: [주니어/미드/시니어]

### 평가 요약
| 평가 축 | 등급 | 핵심 근거 (원문 인용 포함) |
|---------|------|------------------------|
| 문제 인식 | [A/B/C/D] | "[인용]" — [분석] |
| 해결 과정 | [A/B/C/D] | "[인용]" — [분석] |
| 성과/임팩트 | [A/B/C/D] | "[인용]" — [분석] |

### 강점
- [구체적 강점]: "[근거 인용]"

### 개선 필요 사항
- [개선점]: [구체적 개선 방법]

### 신뢰도
| 주요 주장 | 신뢰도 | 근거 |
|----------|--------|------|
| "[인용]" | 근거 충분/부분적/불충분 | [판단 이유] |
```

## 평가 시 주의사항

1. **가혹하되 건설적이어야 한다**: 문제를 지적할 때 반드시 구체적 개선 방법을 제시
2. **맥락을 고려하라**: 주니어/시니어, 회사 규모, 도메인에 따라 기대치 조정
3. **원문을 인용하라**: 판단의 근거를 이력서/포트폴리오에서 직접 인용
4. **확인할 수 없는 것은 확인할 수 없다고 밝혀라**: 추측으로 판단하지 않고, 근거 불충분으로 표기
5. **비교 대상을 명시하라**: "좋다/나쁘다"가 아닌, 어떤 기준 대비인지 명확히
