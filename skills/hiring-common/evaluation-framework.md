# 채용 평가 프레임워크

> 실제 기업 채용 프로세스를 기반으로 한 평가 체계.
> ATS 스크리닝, 리크루터 리뷰, FAANG 스타일 면접 루브릭을 참고하여 구성.

---

## 공통 판정 체계

모든 단계에서 동일한 판정 등급을 사용한다:

```
[PASS]   — 다음 단계 진행 가능
[REJECT] — 이 단계에서 탈락. 구체적 피드백과 개선 방향 제공.
```

> NEEDS_WORK 없음. 실제 채용에서는 통과 아니면 탈락이다.
> 대신 REJECT 시 "이 부분을 보완하면 통과 가능" 수준의 상세 피드백을 제공한다.

## 회사 유형별 평가 기준 차등

채용 공고의 회사 유형에 따라 평가 엄격도를 달리한다.

### 회사 유형 분류

| 유형 | 예시 | 분류 기준 |
|------|------|----------|
| 전통 대기업 | 삼성, LG, 현대, SK | 공채/수시 구분, GSAT 등 필기, 자격증 가산 |
| 유니콘/스케일업 | 토스, 카카오, 당근, 쿠팡 | 블라인드 채용, 연차 무관 명시, 코테/과제 중심 |
| 스타트업 | Seed~Series B | 포트폴리오 중심, 빠른 프로세스, 미션 정렬 |
| 글로벌 빅테크 | Google, Amazon, Meta | 레벨 기반, 학위 불필요, 면접 성과 중심 |

> 분류가 불분명하면 공고 키워드로 추론한다:
> "공채", "GSAT", "TOEIC" → 전통 대기업
> "블라인드 채용", "연차 무관", "자유 양식" → 유니콘
> "소규모 팀", "시리즈 A/B", "대표 면접" → 스타트업

### 경력 연차 유연성

공고의 요구 경력과 실제 경력 간 갭 허용 범위:

| 공고 요구 경력 | 전통 대기업 | 유니콘/스케일업 | 스타트업 | 글로벌 빅테크 |
|--------------|-----------|---------------|---------|-------------|
| 신입 (0년) | 0년만 가능 (공채) | 0~1년 | 0~2년 | 0~1년 |
| 3년 | 2~4년 | 1~4년 | 0~4년 (실력 증명 시) | 레벨 기반 판단 |
| 5년 | 4~6년 | 3~6년 | 2~6년 | 레벨 기반 판단 |
| 7년+ | 6~8년 | 5~8년 | 4~8년 | 레벨 기반 판단 |

**경력 갭 점수 반영:**
```
갭 없음 또는 허용 범위 내   → 경력 연차 매칭 점수 100%
허용 범위 경계 (±1년)      → 경력 연차 매칭 점수 70%
허용 범위 초과              → 경력 연차 매칭 점수 30% (대기업은 0%)
```

> 핵심 원칙: 공고의 경력 요건은 "희망 사항"에 가깝다. 특히 유니콘/스타트업에서는
> 실력을 증명할 수 있다면 연차 부족이 탈락 사유가 되지 않는 경우가 많다.
> 다만 전통 대기업의 경력직 채용은 연차 범위를 엄격하게 적용한다.

### 항목별 엄격도 차등

| 평가 항목 | 전통 대기업 | 유니콘/스케일업 | 스타트업 | 글로벌 빅테크 |
|----------|-----------|---------------|---------|-------------|
| 필수 자격 요건 | **엄격** (1개라도 미충족 시 REJECT) | **유연** (핵심 1~2개만 필수) | **매우 유연** (포트폴리오로 대체 가능) | **유연** (면접 성과로 대체) |
| 학력/학교 | 평가 (암묵적 가산) | 무관 (블라인드) | 무관 | 공식 무관 |
| 자격증/어학 | 가산점 또는 최소 기준 | 무관 | 무관 | 무관 |
| 기술 스택 매칭 | 중요 | **매우 중요** (핵심 필터) | 중요하나 학습 의지 고려 | **매우 중요** |
| 포트폴리오/Github | 보조 자료 | **핵심 평가 요소** | **가장 중요한 평가 요소** | 보조 자료 |
| 컬처핏 | 조직 적응력 | **높은 비중** | **최고 비중** | 높은 비중 (LP, Googleyness) |

### ATS Hard Filter 차등 적용

```
전통 대기업:
  - 필수 자격 요건 전체를 Hard Filter로 적용
  - X가 1개라도 있으면 즉시 REJECT
  - 어학 최소 기준 미달 시 REJECT

유니콘/스케일업:
  - "핵심 필수"만 Hard Filter (기술 스택 관련 1~2개)
  - 나머지 필수는 Soft Filter (점수 감점, 즉시 탈락 아님)
  - 학력/자격증은 Hard Filter에서 제외

스타트업:
  - Hard Filter 거의 없음 (기술 스택 핵심 1개 정도)
  - 대부분 Soft Filter (종합 점수로 판단)
  - 포트폴리오/프로젝트 경험이 자격 요건을 대체 가능

글로벌 빅테크:
  - 기술 스택 관련만 Hard Filter
  - 학력/자격증 Hard Filter 없음
  - 경력 레벨(L3/L4/L5)에 맞는 역량 기대치 적용
```

## 1단계: 서류 심사 평가 기준

### 1-1. ATS 시뮬레이션 (자동 스크리닝)

실제 ATS는 이력서를 파싱하여 JD와의 키워드 매칭 점수(0~100)를 산출한다.

**평가 항목:**

| 항목 | 설명 | 비중 |
|------|------|------|
| 필수 자격 요건 | JD의 "자격 요건"과 일치하는 항목 수 | Hard Filter |
| 기술 스택 매칭 | JD에 명시된 기술과 이력서 기술의 일치율 | 40% |
| 경력 연차 매칭 | JD 요구 경력과 실제 경력의 부합 | 20% |
| 직무 관련성 | 이전 직무/프로젝트가 해당 포지션과 관련되는 정도 | 25% |
| 학력/자격증 | JD에 명시된 학력/자격 요건 충족 여부 | 15% |

**Hard Filter (즉시 탈락):**
- 필수 자격 요건 중 하나라도 미충족 시 → 점수 무관 REJECT
- 예: "Java 3년 이상 필수"인데 Java 경험 없음

**점수 기준:**
```
70점 이상 → ATS 통과 (리크루터 리뷰 진행)
70점 미만 → ATS REJECT
```

> 실제 Fortune 500 기업의 75%가 ATS 단계에서 이력서를 걸러낸다.
> 이 시뮬레이션은 사용자에게 ATS 최적화 피드백을 제공하는 데 목적이 있다.

### 1-2. 리크루터 스크리닝 (6초 → 30초 → 2분)

실제 리크루터는 평균 6~8초 내에 1차 판단, 통과 시 30초 심화, 관심 시 2분 상세 검토한다.

**6초 스캔 체크리스트 (즉시 판단):**
- [ ] 현재/최근 직무가 포지션과 관련 있는가?
- [ ] 인지도 있는 회사 경력이 있는가? (판단 편향이지만 현실)
- [ ] 재직 기간이 안정적인가? (3년 내 3회 이상 이직 = Red Flag)
- [ ] 이력서 포맷이 깔끔한가?

**30초 심화 체크리스트:**
- [ ] 경력 공백이 6개월 이상 있는가? (설명 없으면 Red Flag)
- [ ] 경력 궤적이 상승하고 있는가? (직급, 책임 범위 증가)
- [ ] 정량적 성과가 포함되어 있는가?
- [ ] JD 핵심 키워드가 이력서에 있는가?

**2분 상세 검토 (30초 통과 시):**
- [ ] 프로젝트 설명에 구체적 기여와 성과가 있는가?
- [ ] 기술 스택 깊이가 적정한가? (나열만 vs 실제 활용 경험)
- [ ] 팀 규모, 역할이 명시되어 있는가?
- [ ] 면접에서 더 확인하고 싶은 포인트가 있는가?

**판정:**
```
PASS   — 면접 초대 대상 (상위 ~3%)
REJECT — 탈락 (구체적 개선 피드백 제공)
```

## 2단계: 코딩 테스트 평가 기준

Google/Meta 스타일 4차원 루브릭. 각 차원 1~4점.

### 평가 차원

| 차원 | 1점 (Strongly Reject) | 2점 (Reject) | 3점 (Pass) | 4점 (Strong Pass) |
|------|----------------------|--------------|------------|-------------------|
| **문제 해결** | 문제를 이해하지 못함 | 비효율적 해법, 가이드 필요 | 올바른 해법, 최적은 아님 | 여러 해법 비교 후 최적 선택 |
| **코딩 능력** | 코드 작성 불가 | 문법 오류 다수, 기본 구현 미흡 | 동작하는 코드, 소소한 오류 | 깔끔하고 정확한 코드, 전문가 수준 |
| **커뮤니케이션** | 설명 불가 | 불명확, 정리 안 됨 | 적절한 설명 | 논리적이고 명확한 소통 |
| **테스트/검증** | 테스트 없음 | 기본 케이스만 확인 | 엣지 케이스 일부 처리 | 체계적 테스트, 자기 수정 |

**판정:**
```
PASS   — 4개 차원 모두 2점 이상 AND 평균 2.5점 이상
REJECT — 어느 하나라도 1점 OR 평균 2.5점 미만
```

> 실제 FAANG에서는 하나의 차원에서 1점이면 다른 차원 점수와 무관하게 탈락이다.

### 난이도별 기대치

| 경력 | 기대 수준 |
|------|----------|
| 주니어 (0~3년) | 기본 자료구조/알고리즘. 최적해 아니어도 동작하는 코드 |
| 미드 (3~7년) | 최적 해법 도달. 복잡도 분석. 엣지 케이스 처리 |
| 시니어 (7년+) | 최적해 + 트레이드오프 논의 + 확장성 고려 |

### 2-2. Take-home 과제 루브릭

사전 과제(Take-home) 평가 시 사용. 알고리즘/시스템설계/실무 모드와 별도의 루브릭.

| 차원 | 1점 (Strongly Reject) | 2점 (Reject) | 3점 (Pass) | 4점 (Strong Pass) |
|------|----------------------|--------------|------------|-------------------|
| **요구사항 충족** | 핵심 기능 미구현 | 기본 기능만 구현 | 모든 요구사항 충족 | 요구사항 + 추가 개선 |
| **코드 품질** | 구조 없음, 가독성 낮음 | 기본 구조, 개선 여지 많음 | 깔끔한 구조, 적절한 패턴 | 프로덕션 수준 코드 |
| **설계 의사결정** | 설계 근거 없음 | 기본 선택, 대안 미검토 | 합리적 선택 + 일부 근거 | 명확한 근거 + 트레이드오프 |
| **테스트/문서화** | 테스트/문서 없음 | 최소한의 테스트 | 적절한 테스트 커버리지 | 체계적 테스트 + README |

**판정:**
```
PASS   — 4개 차원 모두 2점 이상 AND 평균 2.5점 이상
REJECT — 어느 하나라도 1점 OR 평균 2.5점 미만
```

> take-home 과제는 모범 답안을 제공하지 않는다. 대신 코드 리뷰 형식의 상세 피드백을 제공한다.
> PASS 시 다음 기술 면접에서 제출 코드 기반 질문이 50% 포함된다.

## 3단계: 기술 면접 평가 기준

### 평가 차원 (Google/Meta 하이브리드)

| 차원 | 비중 | 평가 내용 |
|------|------|----------|
| 기술적 깊이 | 30% | 기술 선택 근거, 내부 동작 이해, 트레이드오프 인식 |
| 문제 해결 | 25% | 체계적 접근, 문제 분해, 제약 조건 파악 |
| 시스템 설계 | 25% | 아키텍처 적합성, 확장성, 장애 대응 (시니어 비중 ↑) |
| 커뮤니케이션 | 20% | 설명의 명확성, 논리적 구조, 질문에 대한 응답 |

### 판정 스케일 (6단계)

```
Strong Hire    — 이 사람을 반드시 채용해야 한다
Hire           — 채용을 추천한다
Leaning Hire   — 약간의 우려가 있지만 채용 가능
Leaning Reject — 우려가 있어 채용을 추천하기 어렵다
Reject         — 채용하지 않는다
Strong Reject  — 명백히 부적합하다
```

**판정 기준:**
```
PASS   — Leaning Hire 이상
REJECT — Leaning Reject 이하
```

### 경력별 기대치

| 경력 | 기술 깊이 | 시스템 설계 | 커뮤니케이션 |
|------|----------|-----------|------------|
| 주니어 | 기본 CS + 주력 기술 이해 | 간단한 CRUD 설계 | 질문에 답변 가능 |
| 미드 | 기술 선택 근거 설명 | 중규모 시스템 설계 | 능동적 설명 |
| 시니어 | 트레이드오프 분석 + 대안 비교 | 대규모 분산 시스템 | 비전공자에게도 설명 가능 |

## 4단계: 인성/컬처핏 면접 평가 기준

### STAR 기반 역량 평가 (5점 척도)

| 점수 | 정의 |
|------|------|
| 5 | 풍부한 경험, 다수의 구체적 사례, 측정 가능한 결과 |
| 4 | 다수의 경험, 명확한 주도성과 임팩트 |
| 3 | 표준적 경험, 역량 보유 확인 |
| 2 | 제한적 경험, 모호하거나 불완전한 사례 |
| 1 | 관련 경험 없음 또는 사례 제시 불가 |

### 평가 역량 영역

| 영역 | 평가 포인트 |
|------|-----------|
| 협업/팀워크 | 의견 충돌 해결, 타인과의 협력, 피드백 수용 |
| 주도성/오너십 | 자발적 문제 발견, 역할 범위 초월, "내 일" 의식 |
| 성장 마인드셋 | 실패에서의 학습, 새로운 기술 습득, 자기 개선 |
| 커뮤니케이션 | 복잡한 상황 설명 능력, 이해관계자 관리 |
| 문화 적합도 | 회사 가치와의 정렬 (JD 기반), 업무 방식 호환성 |

**판정:**
```
PASS   — 5개 영역 모두 2점 이상 AND 평균 3점 이상
REJECT — 어느 하나라도 1점 OR 평균 3점 미만
```

## 종합 판정 (통합 파이프라인용)

### 판정 규칙

```
최종 PASS   — 모든 단계 PASS
최종 REJECT — 어느 한 단계라도 REJECT (즉시 중단)
```

### 단계별 통과율 참고 (실제 테크 기업 기준)

| 전환 | 통과율 |
|------|--------|
| 지원 → ATS 통과 | ~25% |
| ATS → 리크루터 통과 | ~12% |
| 전체 지원 → 면접 초대 | ~3% |
| 면접 → 오퍼 | ~27% |
| 전체 지원 → 최종 합격 | ~0.7~1% |

> 이 통과율은 실제 채용 환경의 참고 수치이며, 시뮬레이션에서는 학습 목적으로
> REJECT 후에도 구체적 개선 피드백을 제공한다.

---

## 평가 시 공통 원칙

1. **JD 기준 평가**: 모든 평가는 채용 공고의 요구사항에 근거한다. JD가 없으면 일반적 포지션 기준 적용.
2. **원문 인용**: 판단 근거를 이력서/포트폴리오에서 직접 인용한다.
3. **추측 금지**: 확인 불가 항목은 "확인 불가"로 표기한다.
4. **경력 수준 명시**: 어떤 경력 수준 기준을 적용했는지 밝힌다.
5. **건설적 피드백**: REJECT 시 "왜 탈락인지" + "무엇을 어떻게 고치면 통과 가능한지" 반드시 제시.
