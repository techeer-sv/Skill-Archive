# 통합 채용 파이프라인 (Hiring Full Pipeline)

서류 심사 → 코딩 테스트 → 기술 면접 → 인성 면접까지 전체 채용 프로세스를 단계별로 안내합니다.

---

## 작동 방식

이 스킬은 **가이드 모드**로 동작합니다.
각 단계를 독립적으로 실행하되, 단계 결과를 파일로 저장하여 다음 단계에서 참조합니다.

```
[서류 심사] → 결과 저장 → [코딩 테스트] → 결과 저장 → [기술 면접] → 결과 저장 → [인성 면접] → 종합 리포트
```

> 하나의 세션에서 모든 단계를 완료하기 어려울 수 있습니다.
> 각 단계 결과가 파일로 저장되므로, 다른 세션에서 이어서 진행할 수 있습니다.

---

## 실행 절차

### Step 1: 입력 수집

먼저 `.hiring-pipeline/input-context.md`가 존재하는지 확인한다.
- **존재하면**: Read하여 이전 세션의 입력 정보(이력서 경로, 포트폴리오 경로, 공고 내용)를 복원한다. 사용자에게 "이전 세션의 입력 정보를 발견했습니다. 그대로 사용할까요?" 확인 후, 변경 없으면 Step 2로 건너뛴다. 또한 `.hiring-pipeline/` 내 stage 파일 존재 여부를 확인하여 완료된 단계를 자동 감지하고, 다음 미완료 단계부터 시작을 제안한다.
- **존재하지 않으면**: 아래 절차로 입력을 수집한다.

**이력서 (필수):**
- "이력서 파일 경로를 알려주세요. (PDF, Markdown, TXT)"
- 미제공 시 중단

**포트폴리오 (선택):**
- "포트폴리오가 있으면 파일 경로를 알려주세요. (없으면 '없음')"

**채용 공고 (필수):**
- "채용 공고 내용을 붙여넣어주세요. URL도 가능합니다."
- URL 제공 시 WebFetch 시도. 실패하면 직접 붙여넣기 요청.
- 미제공 시: "통합 파이프라인에서는 채용 공고가 필수입니다. 개별 스킬을 사용하세요: `/hiring-resume-review`, `/hiring-interview`"

**시작 단계:**
- "어떤 단계부터 시작하시겠어요?"
  - "1단계: 서류 심사부터 (처음부터)" — 기본
  - "2단계: 코딩 테스트부터"
  - "3단계: 기술 면접부터"
  - "4단계: 인성 면접부터"

> 중간 단계부터 시작하면 이전 단계 결과 없이 진행됩니다.
> 이전에 저장된 결과 파일이 있으면 자동으로 참조합니다.

**입력 컨텍스트 저장:**

입력 수집이 완료되면 즉시 저장한다:
```
Write 도구로 저장: .hiring-pipeline/input-context.md
```

저장 형식:
```markdown
## 입력 컨텍스트

**이력서 경로:** [절대 경로]
**포트폴리오 경로:** [절대 경로 또는 "없음"]
**채용 공고:** [붙여넣기 내용 또는 "job-analysis.md 참조"]
**시작 단계:** [선택한 단계]
**생성 시각:** [ISO 타임스탬프]
```

### Step 2: 채용 공고 분석 및 저장

공고 내용을 분석하여 다음 형식으로 정리한다:

```markdown
## 채용 공고 분석

**회사:** [회사명]
**포지션:** [포지션명]
**경력:** [요구 경력]

### 필수 요건
- [요건]

### 우대 요건
- [요건]

### 기술 스택
- [기술]

### 업무 내용
- [업무]
```

이 분석 결과를 사용자에게 보여주고 확인한다.

결과를 파일로 저장한다:
```
Write 도구로 저장: .hiring-pipeline/job-analysis.md
```

### Step 3: 파이프라인 실행

각 단계를 순차적으로 실행한다.

---

#### 단계 1: 서류 심사

다음 파일을 Read 도구로 읽고 해당 절차를 수행한다:
- 프로젝트 내: `skills/hiring-resume-review/SKILL.md`
- 설치 환경: `~/.claude/skills/hiring-resume-review/SKILL.md`

> 두 경로 중 존재하는 쪽을 사용한다. 둘 다 없으면 아래 "핵심 판정 기준" 인라인 참조.

**수행 범위:** SKILL.md의 Step 2~5 (입력은 이미 수집됨)

**포트폴리오 제공 시:**
- `hiring-portfolio-review/SKILL.md`도 Read하여 함께 수행
- 이력서와 포트폴리오 **각각 개별 평가** 후, 아래 규칙으로 병합 판정:
  - 3축 각각에서 이력서/포트폴리오 중 **높은 등급**을 채택한다 (포트폴리오가 이력서를 보완)
  - 단, 같은 프로젝트에 대해 이력서와 포트폴리오의 서술이 **모순**되면 별도 언급한다
  - 최종 판정은 병합된 3축 등급으로 산출한다

**결과 저장:**
```
Write 도구로 저장: .hiring-pipeline/stage1-resume-review.md
```

저장 내용: 종합 판정, 평가 요약, JD 매칭, 주요 강점/약점, (포트폴리오 포함 시) 병합 근거

**통과 판정:**
- `PASS` → "서류 심사를 통과했습니다. 다음 단계(코딩 테스트)로 진행할까요?"
- `NEEDS_WORK` → 사용자에게 선택지 제공:
  - "피드백을 반영하여 이력서 수정 후 재제출"
  - "현재 상태로 다음 단계 진행 (참고: 서류 점수가 이후 평가에 반영됩니다)"
  - "여기서 중단하고 개선 작업"
- `REJECT` → 사용자에게 선택지 제공:
  - "피드백을 반영하여 이력서 수정 후 재제출"
  - "여기서 중단하고 개선 작업"

#### 단계 2: 코딩 테스트

이전 단계 결과 참조:
```
Read 도구로 읽기: .hiring-pipeline/stage1-resume-review.md (있는 경우)
Read 도구로 읽기: .hiring-pipeline/job-analysis.md
```

다음 스킬을 Read하여 수행 (프로젝트 내 또는 `~/.claude/skills/` 경로):
- `hiring-coding-test/SKILL.md`

**파이프라인 기본 옵션 (사용자에게 별도 질문하지 않고 자동 적용):**
- 유형: "자동 선택" (서류 심사에서 파악한 포지션 기반)
- 난이도: "자동 (포지션 기반)" (서류 심사에서 파악한 경력 수준 기반)

**컨텍스트 활용:** 서류 심사에서 파악한 기술 수준으로 난이도 자동 결정

**결과 저장:**
```
Write 도구로 저장: .hiring-pipeline/stage2-coding-test.md
```

통과 판정: 단계 1과 동일한 흐름

#### 단계 3: 기술 면접

이전 단계 결과 참조:
```
Read 도구로 읽기: .hiring-pipeline/stage1-resume-review.md (있는 경우)
Read 도구로 읽기: .hiring-pipeline/stage2-coding-test.md (있는 경우)
Read 도구로 읽기: .hiring-pipeline/job-analysis.md
```

다음 스킬을 Read하여 수행 (프로젝트 내 또는 `~/.claude/skills/` 경로):
- `hiring-interview/SKILL.md` (기술 면접 유형으로 실행)

**파이프라인 기본 옵션:**
- 면접 유형: "기술 면접"
- 진행 모드: "학습 모드" (파이프라인에서는 각 단계별 성장을 돕기 위해 학습 모드 기본 적용)

**컨텍스트 활용:**
- 서류에서 발견된 모호한 점 → 면접 딥다이브 대상
- 코딩 테스트 풀이 방식 → 기술적 깊이 질문에 반영

**결과 저장:**
```
Write 도구로 저장: .hiring-pipeline/stage3-tech-interview.md
```

```
Read 도구로 읽기: .hiring-pipeline/stage3-tech-interview.md (있는 경우)

```

다음 스킬을 Read하여 수행 (프로젝트 내 또는 `~/.claude/skills/` 경로):

**파이프라인 기본 옵션:**
- 면접 유형: "인성/컬처핏 면접"

**컨텍스트 활용:**

### Step 4: 종합 리포트

모든 단계(또는 중단 시점까지)의 결과 파일을 Read하여 종합 리포트를 작성한다.

#### 종합 합격 판정 산출 규칙

```
[PASS]       - REJECT가 0개 AND NEEDS_WORK가 1개 이하
[NEEDS_WORK] - REJECT가 0개 AND NEEDS_WORK가 2개 이상
             - 또는 REJECT가 1개 AND NEEDS_WORK가 0개 (나머지 모두 PASS)
             - 또는 REJECT가 1개 AND NEEDS_WORK가 1개 이상
[REJECT]     - REJECT가 2개 이상
             - 또는 서류 심사가 REJECT (기본 자격 미달)
```

> 단, 사용자가 학습 목적으로 REJECT 후에도 계속 진행한 경우,
> "학습 참고용 결과"로 표기하고 위 규칙을 그대로 적용한다.

```markdown
# 채용 프로세스 종합 리포트

## 지원 정보
- **회사:** [회사명]
- **포지션:** [포지션명]
- **진행 단계:** [완료 단계 수] / 4

## 단계별 결과 요약

| 단계 | 판정 | 핵심 피드백 |
|------|------|------------|
| 서류 심사 | [PASS/NEEDS_WORK/REJECT] | [1줄 요약] |
| 코딩 테스트 | [PASS/NEEDS_WORK/REJECT] | [1줄 요약] |
| 기술 면접 | [PASS/NEEDS_WORK/REJECT] | [1줄 요약] |
| 인성 면접 | [PASS/NEEDS_WORK/REJECT] | [1줄 요약] |

## 종합 합격 판정: [PASS / NEEDS_WORK / REJECT]

### 판정 근거
[4단계 결과를 종합한 판정 근거]

## 강점 Top 3
1. [강점]: [근거]
2. [강점]: [근거]
3. [강점]: [근거]

## 약점 Top 3
1. [약점]: [근거] → [개선 방법]
2. [약점]: [근거] → [개선 방법]
3. [약점]: [근거] → [개선 방법]

## 포지션 적합도

| 요건 | 충족 | 근거 |
|------|------|------|
| [필수 요건 1] | O/X/△ | [어느 단계에서 확인] |

### 종합 적합도: [매우 적합 / 적합 / 보통 / 부적합]

## 개선 로드맵

### 단기 (1~2주)
- [즉시 개선 가능한 항목]

### 중기 (1~3개월)
- [학습/경험 필요 항목]

### 장기 (3개월+)
- [커리어 방향성 제안]
```

종합 리포트도 저장한다:
```
Write 도구로 저장: .hiring-pipeline/final-report.md
```

## 파일 기반 상태 관리

```
.hiring-pipeline/
  input-context.md             # 입력 정보 (이력서 경로, 포트폴리오 경로, 공고)
  job-analysis.md              # 공고 분석 결과
  stage1-resume-review.md      # 서류 심사 결과
  stage2-coding-test.md        # 코딩 테스트 결과
  stage3-tech-interview.md     # 기술 면접 결과
  stage4-culture-interview.md  # 인성 면접 결과
  final-report.md              # 종합 리포트
```

이 파일들은 다음 용도로 사용된다:
1. **단계 간 컨텍스트 전달**: 이전 단계 결과를 다음 단계에서 Read
2. **세션 간 이어하기**: 다른 세션에서 `.hiring-pipeline/` 파일을 Read하여 중단된 곳부터 재개
3. **사용자 기록**: 파이프라인 완료 후 전체 결과를 보관

## 주의사항

- 각 단계는 개별 스킬과 동일한 품질로 수행된다
- 단계별 결과가 REJECT여도 사용자 선택에 따라 계속 진행 가능 (학습 목적)
- 중간 단계부터 시작 시, 이전 단계 결과 파일이 없으면 해당 컨텍스트 없이 진행
- 전체 파이프라인은 상당한 시간이 소요된다. 단계별로 나눠서 진행하는 것을 권장한다
